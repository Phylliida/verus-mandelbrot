# Session 2026-04-14: Direct Mode, Deep Zoom Precision, GPU Optimizations

## Summary

Massive session covering direct computation fallback, deep zoom precision debugging, GPU kernel optimizations, and transpiler bug fixes. Found and fixed the root cause of zoom noise (`0xFFFF` hex literal parsed as `0` in the transpiler).

## Changes Made

### 1. Direct Computation Fallback
- **`direct_computation_fallback`** helper function: Z_{k+1} = Z_k^2 + c using thread-local arrays
- Fixes pink 16x16 tiles at Mandelbrot boundary (unresolved glitched pixels)
- Verified: 9-op iteration step + escape check, all preconditions proven

### 2. Perturbation Toggle
- `params[6+n] = use_perturbation` (0 = direct only, 1 = perturbation)
- Refinement loop guarded by `use_perturbation != 0u32`
- Web UI checkbox, default checked

### 3. Multi-N Shader Generation
- `generate_shaders.sh`: produces WGSL for n=4,8,16,32,64
- Each shader has compile-time array sizes matching target N
- JS loads correct shader per N, creates pipelines lazily
- N slider extended to 2^6=64

### 4. GPU-Side c Computation
- **Root cause of zoom noise**: per-pixel c values computed on CPU had only ~32 bits of effective precision (all lower limbs zero) because power-of-2 zoom produces clean pixel steps
- **Fix**: pass center + pixel_step as uniforms in params buffer. GPU computes `c = center + (gid - width/2) * step` using verified `signed_mul_to` + `signed_add_to`
- Each pixel gets full n-limb precision from the schoolbook multiplication's carry propagation
- Added `scalar_mul_int` for O(n) offset multiply (vs O(n^2) schoolbook)

### 5. Transpiler Bugs Fixed
- **Hex literal parsing** (`0xFFFF` → `0`): `i64::parse()` doesn't handle hex. Fixed with `from_str_radix` for `0x`, `0b`, `0o` prefixes. This was the root cause of ALL multi-precision arithmetic being wrong on GPU.
- **Bare `{ }` blocks silently dropped**: added `"block"` handler in `parse_block`
- **`return expr;` emitted as bare `return;`**: fixed in both parser (`parse_expr_to_stmt`) and emitter (conditional `return _ret;` vs `return;`)
- **Inline comments in call args parsed as variable names**: removed comment from call site

### 6. Zoom Precision Fixes
- **`doubleToBigFixed` limb ordering**: was placing MSB limb at LSB position (`32*i` → `32*(fracLimbs-1-i)`)
- **BigInt center rescaling**: when switching N, shift centerReBig by `32*(new_frac - old_frac)` bits
- **Max iters decoupled from shared memory**: direct mode allows up to 4096 iters regardless of N

### 7. GPU Kernel Optimizations
- **Re²/im² reuse across escape check** (3 muls/iter instead of 5 = 40% fewer multiplies): escape check computes re²/im² which are reused as ops 1-2 of the next iteration step
- **Periodicity detection**: saves checkpoint every 2^k iterations (8 to 1024), detects exact cycle matches → early exit for in-set pixels
- **Scalar multiply**: O(n) `scalar_mul_int` for GPU c computation instead of O(n^2) schoolbook
- **First iteration Z₁=c by copy**: skip the first multiplication entirely

### 8. GPU Profiler
- Updated `profile_shader.rs` for perturbation kernel (new buffer layout, entry point, params)
- NixOS Vulkan setup documented in README (LD_LIBRARY_PATH + VK_ICD_FILENAMES)
- Baseline profiling results captured

### 9. Dependency Fixes
- `verus-cutedsl`: simplified `arith_add/mul_normalized` specs to match exec (deep flattening → simple swap)
- `verus-fixed-point`: bumped `signed_mul_to_buf` rlimit to 80

### What's Done (Session 2: April 14 continued)

#### Carry Bug Fix
Found and fixed a critical bug: step 4's half×half schoolbook dropped `asum_carry` and `bsum_carry` from step 3. Added step 4b with branchless carry correction using `select_limb`:
- `ca * b_sum` added at offset half via `select_limb(&asum_carry, zero, b_sum[k])`
- `cb * a_sum` added similarly
- `z1_overflow = cc1 + cc2 + ca*cb` tracks the extra limb at position n

#### Helper Function: `add_inplace_propagate`
In `limb_ops_proofs.rs` (27 verified, 0 errors). Adds b[0..n] to out[start..start+n] in-place, adds overflow at position n, propagates carry through tail positions. Uses `lemma_vec_val_set_one` for per-element value tracking (avoids sem_seq unfolding issues). Postcondition:
```
vec_val(new_region) + carry * P(n+tail) == vec_val(old_region) + vec_val(b) + overflow * P(n)
```

#### Value Equation Tracking through Sub-Borrow Loops
Both sub_borrow loops (step 5) now track the value equation:
- Loop 1: `vec_val(scratch_region) == z1_full_n_val - vec_val(z0_partial) + borrow1 * P(i)`
- Loop 2: `vec_val(scratch_region) == scratch_post_sub1_val - vec_val(z2_partial) + borrow2 * P(i)`
Each proved using `lemma_vec_val_set_one` + `lemma_vec_val_split` + `nonlinear_arith`.

#### Overflow Bound Lemma
`lemma_karatsuba_z1_overflow_bound` in `limb_ops_proofs.rs` (verified). Proves `z1_overflow - bw1 - bw2` is 0 or 1 given:
- `z1_full >= z0 + z2` (cross terms non-negative)
- `z1_full < z0 + z2 + 2*P` (cross terms bounded)
- Combined sub_borrow value equation

### What's Left (3 Errors, All Cascade from One)

**Root cause: need `z1_full_val = (a_lo_v + a_hi_v) * (b_lo_v + b_hi_v)`**

This connects the computed z1_full (n limbs + overflow) to the algebraic product. Once established, all 3 remaining errors resolve in chain:

1. `lemma_karatsuba_z1_overflow_bound` precondition → fires, proves `ov_diff ∈ {0,1}`
2. Sub_borrow unwrapping → proves `z1_final_overflow.sem() == ov_diff`
3. `add_inplace_propagate` call → satisfies `overflow.sem() == 0 || overflow.sem() == 1`

**To establish z1_full_val, need value tracking through steps 3-4-4b:**

#### Step 3: a_sum/b_sum value equations
Add value equation invariants to the step 3 add3 loop:
```
vec_val(a_sum[0..k]) + asum_carry * P(k) == vec_val(a_lo[0..k]) + vec_val(a_hi[0..k])
```
At k=half: `vec_val(a_sum) + asum_carry * B == vec_val(a_lo) + vec_val(a_hi)`

#### Step 4: schoolbook value equation  
Add value equation invariant to the manual schoolbook loop:
```
vec_val(z1_full_n[0..n]) == vec_val(a_sum) * vec_val(b_sum)  (at end of step 4)
```
This is the same pattern as `mul_schoolbook_to`'s proven postcondition.

#### Step 4b: carry correction value equation
After adding ca*b_sum and cb*a_sum at offset half + z1_overflow at position n:
```
z1_full_n_val + z1_overflow * P == (vec_val(a_sum) + ca*B) * (vec_val(b_sum) + cb*B)
```
This is algebraic expansion + the carry correction loop effects.

#### Final connection
```
(vec_val(a_sum) + ca*B) * (vec_val(b_sum) + cb*B) == (a_lo_v + a_hi_v) * (b_lo_v + b_hi_v)
```
Follows from the step 3 value equations.

#### Frame condition + output postcondition
Once `add_inplace_propagate` call succeeds, its postconditions give:
- Valid limbs on output region
- Frame on non-output region
- Value equation connecting output to z0 + z1*B + z2*B²
The Karatsuba identity then closes `vec_val(output) == a*b`.
